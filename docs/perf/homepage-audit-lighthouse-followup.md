# Homepage performance audit — follow-up

**Date**: 2026-03-15
**Build**: Next.js 16.1.6 (Turbopack), static export
**Scope**: Homepage (`/`) only — Lighthouse/PageSpeed residual issues

---

## 1. Executive summary

| Category | Verdict |
|---|---|
| **Main controllable bottleneck** | Image manifest JSON (~47 KB minified) inlined into page client JS chunk via `"use client"` component imports |
| **Secondary controllable issue** | Several homepage sections are `"use client"` that could potentially be server components or have their client boundaries narrowed |
| **Vendor baseline tax** | GTM + gtag/js + CookieScript JS — ~3 external scripts, no site-side fix possible |
| **Low priority / noise** | Polyfill chunk (110 KB, Next.js default), render-blocking CSS (Tailwind monolithic output, standard), preconnects (marginally premature but harmless) |

**Bottom line**: the single highest-ROI controllable fix is preventing the full image manifest from being bundled into client JS. Everything else is either vendor baseline or standard framework cost.

## 2. Findings by priority

### Finding 2.1 — Image manifest inlined in client JS

- **Severity**: High
- **Category**: JS
- **Lighthouse symptom**: `56d87152cf9e9b9b.js` (104 KB) flagged as unused JS; main-thread work during hydration
- **Root source in repo**:
  - `components/services-section.tsx` — `"use client"`, imports `{ buildSrcSet, getFallbackSrc }` from `@/lib/responsive-image`
  - `components/process-section.tsx` — `"use client"`, imports `ResponsiveImage` from `@/components/responsive-image`
  - `lib/responsive-image.ts` — line 1: `import manifest from "@/data/image-manifest.json"`
  - `data/image-manifest.json` — 70 KB source, ~47 KB minified
- **Build artifact evidence**: `56d87152cf9e9b9b.js` — first ~47,000 chars are the serialized image manifest JSON
- **Why it happens**: When a `"use client"` component imports any module, that module and all its transitive imports become part of the client bundle. `services-section.tsx` directly imports `buildSrcSet`/`getFallbackSrc`, and `process-section.tsx` imports `ResponsiveImage` (which itself imports the same lib). Both paths pull the entire `image-manifest.json` (~3900 lines, all project images) into the client JS chunk. The manifest data is only needed at render time for `srcSet` generation — on the server side this works fine, but the client re-execution pulls it too.
- **Controllability**: Controllable
- **Risk of changing later**: Low — refactoring `services-section.tsx` to receive image props from a server parent or moving `srcSet` computation to server side would not affect functionality

### Finding 2.2 — Homepage client component surface area

- **Severity**: Medium
- **Category**: JS
- **Lighthouse symptom**: hydration cost, main-thread long tasks during page load
- **Root source in repo**: 10 `"use client"` components load on homepage (see section 4)
- **Build artifact evidence**: `56d87152cf9e9b9b.js` (104 KB page chunk) + `add18453ee83df71.js` (30 KB app chunk)
- **Why it happens**: Several homepage sections are `"use client"` for interactive features (tabs, accordion, SVG hover map, drag carousel). Some of these are below-the-fold and could potentially be split or lazy-loaded.
- **Controllability**: Partially controllable — some components genuinely need client interactivity, others might have narrower boundaries
- **Risk of changing later**: Medium — requires careful refactoring to maintain UX

### Finding 2.3 — Render-blocking CSS (Tailwind monolithic output)

- **Severity**: Medium
- **Category**: CSS
- **Lighthouse symptom**: 2 render-blocking CSS resources in critical path
- **Root source in repo**: `app/globals.css` (209 lines) → Tailwind v4 build → `5a4524324a64ef8b.css` (140 KB); `next/font/google` Inter → `4e1c0da8ee9de505.css` (2 KB)
- **Build artifact evidence**: both loaded as `<link rel="stylesheet" precedence="next">` in `<head>`
- **Why it happens**: Tailwind v4 compiles all used utility classes into a single CSS file. This is standard Tailwind behavior — there is no per-route CSS splitting. The 140 KB file contains: Tailwind properties fallback layer, all design tokens (colors incl. lab() variants, spacing, typography), all utility classes used across the entire site, and custom `@layer components` classes from globals.css. The 2 KB file is Inter font `@font-face` declarations (7 unicode-range subsets) generated by Next.js.
- **Controllability**: Partially controllable — critical CSS extraction is possible but complex with Tailwind v4; the font CSS is controlled by Next.js internals
- **Risk of changing later**: High — CSS splitting with Tailwind v4 is not a standard supported workflow

## 3. Render-blocking CSS deep dive

### Chunk mapping

| Chunk | Size | Source | Contents |
|---|---|---|---|
| `5a4524324a64ef8b.css` | 140 KB | `app/globals.css` → Tailwind v4 | Tailwind properties fallback, design tokens (`:root` CSS vars, lab() colors for all Tailwind palette colors), all utility classes used across entire site, custom `@layer components` (section-header, btn-primary, btn-hero, card-premium, glass-card, etc.), CookieScript badge hiding media query, `.no-scrollbar` utility, `@keyframes` (spin, ping, pulse), `@property` declarations for all Tailwind CSS custom properties |
| `4e1c0da8ee9de505.css` | 2 KB | `next/font/google` (Inter) in `app/layout.tsx:2` | 7 `@font-face` declarations with unicode-range subsetting (Cyrillic ext, Cyrillic, Greek ext, Greek, Vietnamese, Latin ext, Latin) + 1 fallback `@font-face` (Arial with metric overrides) + generated CSS module class |

### Why the small CSS file exists separately

The font CSS is generated by Next.js `next/font` system at build time as a separate chunk. It cannot be merged with the Tailwind output because they go through different compilation pipelines:
- Tailwind CSS: `app/globals.css` → `@tailwindcss/postcss` → single output
- Font CSS: `next/font/google` → Next.js internal font optimizer → separate chunk

This is a Next.js framework constraint, not a project configuration issue.

### Best candidate for future fix

The large Tailwind chunk (140 KB) is the only realistic optimization target, but options are limited:
- **Critical CSS extraction**: extract above-the-fold CSS and defer the rest. Complex with Tailwind v4, no standard tooling.
- **Reduce Tailwind surface**: the `@source not "../components/ui"` directive already excludes shadcn/ui components. Further reduction would require auditing utility usage across all pages.
- **Note**: The 140 KB is uncompressed. With gzip/brotli this is typically ~15-20 KB over the wire. The render-blocking nature is the real issue, not the transfer size.

## 10. Recommended next-step queue

### 1. Fix image manifest client bundle leak (High ROI)

**Target**: prevent `data/image-manifest.json` from being inlined in client JS.
**Approach options**:
- Refactor `services-section.tsx` to receive pre-computed `srcSet`/`src` as props from a server wrapper component
- Same for `process-section.tsx` — pass image props from server parent
- This removes ~47 KB from the 104 KB page chunk (~45% reduction)
**Risk**: Low — functional behavior unchanged

### 2. Review client component boundaries (Medium ROI)

**Target**: reduce hydration surface on homepage.
**Candidates in order of impact**:
- `FaqSection` (116 lines) — could use native `<details>`/`<summary>` instead of React state
- `ProcessSection` (240 lines) — accordion pattern, same native HTML opportunity
- `WorkAreaSection` (296 lines) — below fold, could be `dynamic()` imported
**Risk**: Medium — requires testing each component's UX after change

### 3. Evaluate Tailwind CSS critical extraction (Low ROI, high effort)

**Target**: reduce render-blocking CSS.
**Reality**: Tailwind v4 does not support per-route splitting. Critical CSS extraction requires custom tooling (e.g., Critters). The 140 KB file compresses to ~15-20 KB with brotli. Likely not worth the complexity for the marginal FCP improvement.

### What should be left alone

- **GTM/GA4 integration** — single clean integration, no optimization possible from site side
- **CookieScript loading** — vendor baseline, correctly loaded via GTM
- **Polyfill chunk** — Next.js framework artifact, loaded low-priority, compresses well
- **Preconnects** — marginally premature but cost is near-zero; removing them risks slower loads if timing changes
- **ETICS images** — well-optimized, remaining savings are binary-only (~5-10 KB total across all 5 images)
- **Font CSS chunk** — 2 KB, Next.js internals, not actionable
